老天……这段代码真是tricky得可以啊。我想我有必要解说一下。首先贴上正确的文字版：


(*我加了个N@ 来提速*)
c = #\[Conjugate]
ArrayPlot@
Log[N@BinCounts[
Through@{Re, Im}@# & /@ NestList[(5 # c + Re@#^6 - 2.7) # + c^5 &, .1 + .2 I, 9^7],
a = {-1, 1, 0.001}, a] + 1] // AbsoluteTiming


作者在纯函数外面定义了一个代码块，用来节省代码，这种写法通常而言，是错误的，举个例子：
part = #; wrong = part &
wrong@3
(* part & *)
(* #1 *)
wrong[3]是无法输出3的，因为#被包在part里面，没有在纯函数中“显式存在”。这个问题在这篇帖子（http://tieba.baidu.com/p/4886024974）里已经说得很多了，这里不再多讲。但是，在这个例子里，c=Conjugate[#]却被正确地“识别”了，因为作者巧妙地运用了几个trick。


首先，Nest/NestList等函数是可以自动编译的，并且，在自动编译时，会加上选项CompilationOptions -> {"InlineExternalDefinitions" -> True}，关于这个可以部分地参看这帖：http://mathematica.stackexchange.com/q/42419/1871


然后，CompilationOptions -> {"InlineExternalDefinitions" -> True}又有一个不为人知的性质（我是直到几分钟前才发现），通常，这一选项只可以inline可编译的数值常数项，也就是，只能让形如


<<CompiledFunctionTools`
a = 2;
Compile[{c}, a c,
CompilationOptions -> {"InlineExternalDefinitions" -> True}] // CompilePrint


的函数自动编译，至于
a=#;
因为#仅仅是个符号，通常是不能编译的，比如：
a = #;
Compile[{c}, a &@c,
CompilationOptions -> {"InlineExternalDefinitions" -> True}] // CompilePrint
但是，这里出现了一个例外，那就是，如果含了外部表达式的纯函数是Nest/NestList的第一个参数，并且那个外部表达式中所含的非数值符号仅限#的话，a将是可以inline的！比如：
a = #;
Compile[{c}, Nest[a &, c, 10],
CompilationOptions -> {"InlineExternalDefinitions" -> True}] // CompilePrint
(*完全编译*)
注意这招仅对#有效：
a = Conjugate@b;
Compile[{c}, Nest[Function[b, a + b], c, 10],
CompilationOptions -> {"InlineExternalDefinitions" -> True}] // CompilePrint
(* 行不通 *)

从另外一个角度看, Function生成的函数(纯函数的标准形式)本身在求值时, 是遵循之前 @xzcyr 说过的"显式存在", 以及参数列表要显式存在, 才会进行求值. 而Function本身的HoldAll特点, 让内部的Slot变量s1并不能立即计算出来, 所以形式就被Hold下来了


如下示例:


Function[{x}, Sin[x]][Pi]
Function[Sin[Slot[1]]][Pi]
s1 = Slot[1];
Function[Sin[s1]][Pi]
